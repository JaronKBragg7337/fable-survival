## Live deploy model

Fable has two public surfaces:

1. **Standalone canonical app:** https://fable-survival.vercel.app
2. **Heartbeat mirror:** https://www.heartbeatobservatory.com/games/fable-survival/

The standalone app deploys automatically from this repo's `main` branch via
Vercel Git integration.

The Heartbeat mirror deploys from the separate
`JaronKBragg7337/heartbeat-observatory` repo. To keep it current, a computer-
capable agent must build this repo and copy the generated `dist/` files into
`heartbeat-observatory/games/fable-survival/`, while preserving Heartbeat-only
wrapper behavior.

This mirrors the SYL workflow: Fable source stays here; Heartbeat hosts a static
copy under `/games/fable-survival/`.

## Recovery procedure (new machine, from zero)

1. **Install tooling** (all free):
   - Git: https://git-scm.com
   - Node.js LTS (18+): https://nodejs.org
   - GitHub CLI: https://cli.github.com
   - Vercel CLI: `npm i -g vercel` (or use `npx vercel`)

2. **Authenticate** (owner does these interactive logins):
   ```bash
   gh auth login          # browser flow, GitHub account JaronKBragg7337
   vercel login           # browser flow, Vercel account kylerbragg73-2101
   ```

3. **Clone and set identity:**
   ```bash
   gh repo clone JaronKBragg7337/fable-survival
   cd fable-survival
   git config user.name  "JaronKBragg7337"
   git config user.email "JaronKBragg7337@users.noreply.github.com"
   ```

4. **Install and verify the build:**
   ```bash
   npm install
   npm run build      # must succeed; if rollup optional-dep error: delete
                      # node_modules + package-lock.json and npm install again
                      # (package-lock is platform-sensitive for rollup binaries)
   ```

5. **Relink Vercel and verify standalone deploy, if needed:**
   ```bash
   vercel link --yes --project fable-survival
   vercel deploy --prod --yes     # from PROJECT ROOT, never from dist/
   ```

6. **Verify end-to-end:** open https://fable-survival.vercel.app, check console
   for errors, submit a 💬 feedback or 🤖 dev-request, confirm the issue appears:
   ```bash
   gh issue list --label player-feedback --state open
   ```

7. Continue development per CLAUDE.md / AGENTS.md.

## Heartbeat mirror deploy

Run this after every finished/verified Fable change unless the owner explicitly
asks for a standalone-only experiment.

### Boundaries

- Fable repo is canonical for game code and behavior.
- Heartbeat repo is the static hosted mirror and website shell.
- Do not touch Heartbeat engine, shell, `hb-device-tier.js`, SYL, Kimi expansion
  work, Social, Library, Theater, PAM, Supabase schema, or unrelated website
  files while syncing Fable.
- Preserve/reapply Heartbeat-only wrapper behavior in
  `heartbeat-observatory/games/fable-survival/index.html`, especially:
  - `<script src="/hb-device-tier.js"></script>` in the head;
  - the hosted graphics/quality chip, if still present and not breaking current UI;
  - any Heartbeat-only hosted-path shell behavior.
- Feedback and 🤖 dev chat must remain no-credit request inboxes. They create
  GitHub issues labeled `player-feedback`; they do not call Anthropic, OpenAI,
  Perplexity, Grok, or any other AI model API.

### Procedure

```bash
# 1. Clone/pull canonical Fable
cd /path/to/workspace
gh repo clone JaronKBragg7337/fable-survival fable-survival 2>/dev/null || true
cd fable-survival
git pull
npm install
npm run build

# 2. Clone/pull Heartbeat
cd /path/to/workspace
gh repo clone JaronKBragg7337/heartbeat-observatory heartbeat-observatory 2>/dev/null || true
cd heartbeat-observatory
git pull

# 3. Inspect current hosted wrapper before replacing
cp -R games/fable-survival games/fable-survival.backup

# 4. Replace the built static game output
rm -rf games/fable-survival/assets
cp -R ../fable-survival/dist/* games/fable-survival/
```

Then manually inspect and patch:

```text
heartbeat-observatory/games/fable-survival/index.html
```

Required wrapper checks:

- Head contains `/hb-device-tier.js`.
- Current Fable UI is present: 💬 feedback on start/death screens and 🤖
  dev-request inbox panel.
- Heartbeat-only quality chip/script is preserved if still desired.
- Asset script points to the new copied `./assets/...js` file from the latest build.

### API route check for Heartbeat mirror

The Heartbeat-hosted page is served from:

```text
https://www.heartbeatobservatory.com/games/fable-survival/
```

So browser calls to `/api/feedback` and `/api/aichat` hit Heartbeat's domain, not
standalone Fable's domain. Therefore the Heartbeat repo must include compatible
routes:

```text

heartbeat-observatory/api/aichat.js
```

These routes should create issues in:

```text

```

with label:

```text

```

They must not call any AI model API. If the server env secret is missing, they
should fail gracefully with `not-configured` instead of crashing the game.

## Disaster scenarios

- **MSI dies:** follow recovery above. Zero code/data loss.
- **Standalone Vercel account lost:** clone repo, `npm run build`, deploy project
  root to a new static+functions host. Recreate the issue-inbox env secret there.
  Update README/CLAUDE with new URL.
- **Heartbeat mirror drifts:** use `HEARTBEAT_SYNC_PROMPT.md` with a computer-
  capable agent. Rebuild Fable, copy `dist/` into Heartbeat, preserve Heartbeat
  wrapper pieces, push Heartbeat.
- **GitHub account lost:** the full repo with history exists as a clone on any
  dev machine; push it to a new remote. Vercel keeps serving the game meanwhile.
  Feedback/dev-chat pipelines need the new repo path constants + new issue-inbox
  secret.
- **Both lost simultaneously:** any machine with a clone still holds complete
  source + history. This is why at least one up-to-date clone should always exist
  somewhere.
