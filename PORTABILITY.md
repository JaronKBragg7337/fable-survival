# PORTABILITY.md — Full Recovery on a Brand-New Machine

Purpose: the original development laptop ("the MSI") must never be a single
point of failure. This document is the complete, tested recovery path. If the
MSI vanished today, a human + any AI agent can rebuild the entire development
environment from GitHub + the owner's accounts in ~15 minutes.

passwords + 2FA), not the laptop.

## Recovery procedure (new machine, from zero)

1. **Install tooling** (all free):
   - Git: https://git-scm.com
   - Node.js LTS (18+): https://nodejs.org
   - GitHub CLI: https://cli.github.com
   - Vercel CLI: `npm i -g vercel` (or use `npx vercel`)

2. **Authenticate** (owner does these two interactive logins):
   ```bash
   gh auth login          # browser flow, GitHub account JaronKBragg7337
   vercel login           # browser flow, Vercel account kylerbragg73-2101
   ```

3. **Clone and set identity:**
   ```bash
   gh repo clone JaronKBragg7337/fable-survival
   cd fable-survival
   git config user.name  "JaronKBragg7337"
   git config user.email "JaronKBragg7337@users.noreply.github.com"
   ```

4. **Install and verify the build:**
   ```bash
   npm install
   npm run build      # must succeed; if rollup optional-dep error: delete
                      # node_modules + package-lock.json and npm install again
                      # (package-lock is platform-sensitive for rollup binaries)
   ```

5. **Relink Vercel and verify deploy:**
   ```bash
   vercel link --yes --project fable-survival
   vercel deploy --prod --yes     # from PROJECT ROOT, never from dist/
   ```

6. **Verify end-to-end:** open https://fable-survival.vercel.app, check console
   for errors, submit a 💬 feedback, confirm the issue appears:
   ```bash
   gh issue list --label player-feedback --state open
   ```

7. Continue development per CLAUDE.md / AGENTS.md.

## Secrets (NEVER commit these — recreate instead)

**GITHUB_TOKEN** (used by /api/feedback to create issues):
- Already stored in Vercel cloud env vars, so it survives laptop loss.
- To recreate (expiry, leak, or new Vercel project):
  1. GitHub → Settings → Developer settings → Fine-grained personal access
     tokens → Generate new token
  2. Repository access: ONLY `fable-survival`
  3. Permissions: Issues → **Read and write** (Metadata read-only auto-adds).
     Nothing else.
  4. Expiration: owner's choice; note the date in ROADMAP.md recurring section
  5. Owner (never an AI) runs: `npx vercel env add GITHUB_TOKEN production`
     and pastes the token, then redeploy: `vercel deploy --prod --yes`
- Current token expires **2026-09-30** (tracked in ROADMAP.md + GitHub issue).

There are no other secrets. No API keys in code, no .env files, no databases.

## Disaster scenarios

- **MSI dies:** follow recovery above. Zero data loss.
- **Vercel account lost:** clone repo, `npm run build`, deploy `dist/` +
  `api/` to any static+functions host (Netlify: functions need light porting).
  Create new GITHUB_TOKEN env var there. Update README/CLAUDE with new URL.
- **GitHub account lost:** the full repo (with history) exists as a clone on
  any dev machine; push it to a new remote. Vercel keeps serving the game
  meanwhile. Feedback pipeline needs the new repo path in api/feedback.js
  (REPO constant) + new token.
- **Both lost simultaneously:** any machine with a clone still holds complete
  source + history. This is why at least one up-to-date clone should always
  exist somewhere (the MSI today; optionally a second machine or a periodic
  zip backup of the folder).
